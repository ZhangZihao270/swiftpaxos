# HOT Evaluation Plan (SOSP)

## Overview

Evaluation organized by case study. Each case study demonstrates one HOT-optimal protocol,
with throughput-vs-latency as the baseline experiment plus case-specific experiments
that highlight each protocol's unique characteristics.

All experiments use YCSB as the workload generator, extended with per-operation strong/weak annotations.
Throughput-vs-latency curves are generated by increasing the number of concurrent clients.

---

## Testbed

- **Geo-distributed cloud servers**, 5 replicas

## Baselines

| Type | Systems | Purpose |
|------|---------|---------|
| Strong-only | Vanilla Raft, EPaxos, CURP | Upper bound on strong op performance; show benefit of adding weak ops |
| Existing hybrid | MongoDB tunable, Pileus  | Compare with existing hybrid systems |

## Common Config

All experiments share these base settings unless overridden:

```
keySpace: 1000000
reqs: 3000
pendings: 15
pipeline: true
syncs: 1
zipfSkew: 0
```

Thread counts for throughput-vs-latency sweep: `THREADS=(1 2 4 8 16 32 64 96 128)`

Each configuration is run **3 times** and results are averaged.

Results are stored in a dedicated folder per experiment: `results/eval-<exp>-<date>/`

## Case Study 1: Raft-HT (H+T, leader-based)

**Key message**: Simplest hybrid extension. Strong ops completely unchanged from vanilla Raft. Weak ops halve latency (1 vs 2 WAN RTT).

### Exp 1.1: Throughput vs Latency
- **Setup**: Geo-distributed
- **Workload**: YCSB, 50/50 strong/weak, uniform keys
- **Write ratios**: Two groups — **5% writes** and **50% writes**
- **Metric**: Throughput vs median/p99 latency (separate curves for strong and weak ops)
- **Baselines**: Vanilla Raft, MongoDB tunable, Pileus
- **Expected result**: Weak ops achieve ~1 WAN RTT; strong ops same as vanilla Raft; overall latency improves with more weak ops

#### Execution Plan

**Group 1: 5% writes** — 4 protocols × 8 thread counts × 3 repetitions = 96 runs

| Run | Protocol | Config Overrides | Script |
|-----|----------|-----------------|--------|
| A | Raft-HT | `protocol: raftht`, `writes: 5`, `weakRatio: 50`, `weakWrites: 5` | `exp1.1-raftht.sh` |
| B | Vanilla Raft | `protocol: raft`, `writes: 5`, `weakRatio: 0` | `exp1.1-raft.sh` |
| C | MongoDB tunable | TBD (external system) | `exp1.1-mongo.sh` |
| D | Pileus | TBD (external system) | `exp1.1-pileus.sh` |

**Group 2: 50% writes** — 4 protocols × 8 thread counts × 3 repetitions = 96 runs

| Run | Protocol | Config Overrides | Script |
|-----|----------|-----------------|--------|
| A | Raft-HT | `protocol: raftht`, `writes: 50`, `weakRatio: 50`, `weakWrites: 50` | `exp1.1-raftht.sh` |
| B | Vanilla Raft | `protocol: raft`, `writes: 50`, `weakRatio: 0` | `exp1.1-raft.sh` |
| C | MongoDB tunable | TBD (external system) | `exp1.1-mongo.sh` |
| D | Pileus | TBD (external system) | `exp1.1-pileus.sh` |

**Config notes**:
- `writes` 控制 strong ops 的写比例，`weakWrites` 控制 weak ops 的写比例
- `weakWrites` always equals `writes` (same read/write ratio for strong and weak)
- `weakRatio: 50` → 50% ops are weak, 50% are strong

**Output per run** (per thread count, averaged over 3 repetitions):
- Aggregate throughput (ops/sec)
- Strong ops: P50, P99, avg latency (ms)
- Weak ops: P50, P99, avg latency (ms)
- Strong write / strong read / weak write / weak read breakdown

**Results folder**: `results/eval-exp1.1/<date>/`

### Exp 2.2: Conflict Rate Sweep
- **Setup**: Geo-distributed
- **Workload**: YCSB, 50/50 read/write, 50/50 strong/weak, sweep Zipf skewness
- **Metric**: Throughput, P50/P99 latency (strong and weak)
- **Baselines**: Vanilla EPaxos
- **Expected result**: Low conflict: both strong and weak achieve ~1 WAN RTT via fast path. High conflict: fast path degrades, latency increases due to slow path fallback. Shows leaderless protocol's sensitivity to contention.

#### Execution Plan

Fixed thread count (t=32), sweep `ZIPF_SKEW=(0 0.25 0.5 0.75 0.99 1.2 1.5 2.0)`, 3 repetitions each:

| Run | Protocol | Config Overrides | Script |
|-----|----------|-----------------|--------|
| A | EPaxos-HO | `protocol: epaxosho`, `writes: 50`, `weakRatio: 50`, `weakWrites: 50`, `leaderless: true` | `exp2.2-epaxosho.sh` |
| B | Vanilla EPaxos | `protocol: epaxos`, `writes: 50`, `weakRatio: 0`, `leaderless: true` | `exp2.2-epaxos.sh` |

Total: 2 protocols × 8 skew values × 3 repetitions = 48 runs.

**Output per run**:
- Throughput (ops/sec)
- Strong ops: P50, P99 latency
- Weak ops: P50, P99 latency

**Plot**: X=Zipf skew, Y=throughput (line chart). 2 systems. Secondary plot: X=Zipf skew, Y=P50 latency.

**Results folder**: `results/eval-exp2.2/<date>/`

### Exp 2.3: Failure Recovery (EPaxos-HO vs Raft-HT)
- **Setup**: Geo-distributed
- **Workload**: YCSB, 50/50 read/write, 50/50 strong/weak, zipfian keys
- **Scenario**: Kill one replica (leader for Raft-HT) during steady-state
- **Metric**: Throughput over time
- **Systems**: EPaxos-HO, Raft-HT
- **Expected result**: EPaxos-HO: no election pause (leaderless), remaining majority continues with minimal throughput dip. Raft-HT: leader failure triggers election, brief throughput drop during leader re-election. Demonstrates leaderless vs leader-based failure behavior.

#### Execution Plan

2 runs, long duration (120s), kill at t=60s:

| Run | Protocol | Config Overrides | Script |
|-----|----------|-----------------|--------|
| A | EPaxos-HO | `protocol: epaxosho`, `writes: 50`, `weakRatio: 50`, `weakWrites: 50`, `zipfSkew: 0.99`, `reqs: 100000` | `exp2.3-epaxosho.sh` |
| B | Raft-HT | `protocol: raftht`, `writes: 50`, `weakRatio: 50`, `weakWrites: 50`, `zipfSkew: 0.99`, `reqs: 100000` | `exp2.3-raftht.sh` |

**Failure injection**: At t=60s, `kill -9` replica0 (leader for Raft-HT, any replica for EPaxos-HO).

**Output**:
- Per-second throughput time series (requires instrumentation: client logs timestamp per completed op)
- Recovery time: duration from kill to throughput restoration

**Plot**: X=time (seconds), Y=throughput (ops/sec). 2 lines (EPaxos-HO, Raft-HT). Vertical dashed line at t=60s (failure point).

## Case Study 3: CURP-HO vs CURP-HT (same base, HO vs HT)

**Key message**: Same base protocol, different HOT trade-off. Directly demonstrates the O-vs-T tension from the HOT theorem.

### Exp 3.1: Throughput vs Latency
- **Setup**: Geo-distributed
- **Workload**: YCSB, 50/50 strong/weak, uniform keys
- **Write ratios**: Two groups — **5% writes** and **50% writes**
- **Metric**: Throughput vs median/p99 latency
- **Baselines**: Vanilla CURP
- **Expected result**:
  - CURP-HO: weak ops ~1 LAN RTT (broadcast + nearest ack), strong ~1 WAN RTT fast path
  - CURP-HT: weak writes ~2 WAN RTT (through leader), strong same as vanilla CURP
  - CURP-HO has lower weak latency but lower strong throughput under high weak ratio

#### Execution Plan

**Group 1: 5% writes** — 3 protocols × 9 thread counts × 3 repetitions = 81 runs

| C | Vanilla CURP | `protocol: curp`, `writes: 5`, `weakRatio: 0` | `exp3.1-curp.sh` |

### Exp 3.2: T Property Verification
- **Setup**: Geo-distributed
- **Workload**: YCSB, 50/50 read/write, sweep weak proportion (0%, 25%, 50%, 75%, 99%), zipfSkew=0.99
- **Systems**: CURP-HT, CURP-HO
- **Threads**: t=32 (must be near saturation to reveal T violation through queuing amplification)
- **Metric**: Strong op s_p99 latency as weak proportion increases
- **Expected result**:
  - CURP-HT: s_p99 **decreases** as weak ratio increases (weak ops relieve leader pressure, T satisfied)
  - CURP-HO: s_p99 **increases** at wr=25-50% (weak ops create conflict map entries → strong dep disagreements → slow path + queuing cascade, T violated)
- **This is the key experiment** that empirically validates the T property distinction
- **Why s_p99 not s_p50**: T violation manifests primarily in tail latency. At t=32, s_p50 shows moderate effect (+10-15%), but s_p99 shows dramatic divergence (CURP-HT 5x improvement vs CURP-HO 3-4x degradation)

#### Execution Plan

Fixed thread count t=32, zipfSkew=0.99, sweep `WEAK_RATIOS=(0 25 50 75 99)`, 3 repetitions each:

| Run | Protocol | weakRatio | Config Overrides | Script |
|-----|----------|-----------|-----------------|--------|
| A1-A5 | CURP-HT | 0/25/50/75/99 | `protocol: curpht`, `writes: 50`, `weakWrites: 50`, `zipfSkew: 0.99` | `eval-exp3.2-final.sh` |
| B1-B5 | CURP-HO | 0/25/50/75/99 | `protocol: curpho`, `writes: 50`, `weakWrites: 50`, `zipfSkew: 0.99` | `eval-exp3.2-final.sh` |

Total: 2 protocols × 5 weak ratios × 3 repetitions = 30 runs.

**Results folder**: `results/eval-exp3.2/<date>/`

**Output per run**:
- Strong ops throughput (ops/sec)
- Strong ops P50, P99 latency (ms)
- Weak ops throughput, P50, P99 (for reference)

**Plot 1**: X=weak ratio (%), Y=strong s_p99 latency. 2 lines. CURP-HT decreasing; CURP-HO spike at wr=25-50%.
**Plot 2**: X=weak ratio (%), Y=strong s_p50 latency. 2 lines. CURP-HT stable/decreasing; CURP-HO elevated.

#### Actual Results

**Best configuration for demonstrating T violation**: t=32, zipfSkew=0.99.

At t=32, zipfSkew=0.99 (1 rep):

| WeakRatio | HT tput  | HT s_p50 | HT s_p99 | HO tput  | HO s_p50 | HO s_p99    |
|-----------|----------|----------|----------|----------|----------|-------------|
| 0%        | 17,841   | 71.6ms   | 361.9ms  | 17,312   | 73.0ms   | 416.8ms     |
| 25%       | 21,335   | 72.5ms   | 157.9ms  | 20,122   | 77.0ms   | **1,366.6ms** |
| 50%       | 26,093   | 68.9ms   | 85.0ms   | 27,551   | 83.0ms   | **1,663.5ms** |
| 75%       | 31,103   | 52.3ms   | 78.5ms   | 43,595   | 78.0ms   | 215.7ms     |
| 99%       | 35,038   | 52.9ms   | 78.5ms   | 78,707   | 71.7ms   | 136.7ms     |

**Key findings**:
- CURP-HT s_p50 flat or improving as weakRatio increases → **T property satisfied**
- CURP-HO s_p50 peaks at +14% over baseline at wr50 → **T property violated**
- CURP-HO s_p99 explodes to 1,663ms at wr50 (4x baseline) → **T violation most visible in tail latency**
- At t=8 (unsaturated), both protocols show flat s_p50 — T violation only visible under saturation
